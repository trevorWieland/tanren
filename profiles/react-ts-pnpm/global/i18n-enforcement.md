# i18n Enforcement

No literal user-facing strings in JSX. All user-visible text is rendered through
**`@inlang/paraglide-js`** — a compile-time i18n toolchain that emits typed
message functions per locale and tree-shakes unused messages out of the
production bundle.

```tsx
// ✓ Good: paraglide compiled message functions
import type { ReactNode } from "react";
import * as m from "@/i18n/paraglide/messages";

function WelcomeBanner({ name }: { name: string }): ReactNode {
  return (
    <section>
      <h1>{m.welcomeTitle()}</h1>
      <p>{m.welcomeGreeting({ name })}</p>
      <button type="button">{m.welcomeCta()}</button>
    </section>
  );
}

// ✗ Bad: Hardcoded strings in JSX
function WelcomeBanner({ name }: { name: string }): ReactNode {
  return (
    <section>
      <h1>Welcome</h1>
      <p>Hello, {name}!</p>
      <button type="button">Get Started</button>
    </section>
  );
}
```

**Rules:**
- All user-visible text is rendered through `m.<key>()` calls compiled by
  paraglide-js. Each call is fully typed (parameters and return type) and only
  used messages reach the client bundle.
- No literal strings in JSX. Enforced by oxlint `react/jsx-no-literals` —
  zero violations allowed in CI.
- Allowlist (does not count as a "literal user-facing string"):
  - empty strings (`""`)
  - JSX whitespace and fragments
  - `aria-*` attributes that are NOT user-visible (e.g. role tokens like
    `aria-haspopup="menu"`); user-visible aria text such as
    `aria-label={m.signOutLabel()}` MUST go through paraglide
  - non-text props such as `id`, `name`, `data-*`, `type`, CSS class names
- Interpolation goes through paraglide message parameters
  (`m.greeting({ name })`) — never concatenate translated fragments.

**Layout (single source of truth):**

```
apps/web/src/i18n/
├── project.inlang/
│   └── settings.json          # supported locales + plugin config
├── messages/
│   └── en.json                # source-of-truth catalog (English baseline)
└── paraglide/                 # generated by `pnpm i18n:compile`, gitignored
    ├── messages.js
    ├── messages.d.ts
    └── runtime.js
```

- `messages/<locale>.json` is the **only** place message strings are authored.
  English is the source-of-truth catalog; additional locales mirror its keys.
- The `paraglide/` directory is fully generated and **must** be gitignored.
  Importing from it is the only way to render copy.
- Paraglide ships a typed message function per key:
  `import * as m from "@/i18n/paraglide/messages"` then `m.signUpTitle()`.
  TypeScript surfaces a compile error for missing keys.

**Build pipeline:**
- `pnpm i18n:compile` runs the paraglide compiler against
  `src/i18n/project.inlang/`.
- The script is wired into `pnpm dev` (alongside `next dev` via
  `concurrently`) and `pnpm build` (runs before `next build`) so the
  generated `paraglide/` directory is always fresh before TypeScript or
  Next.js see it.
- `just bootstrap` runs the first compile so a clean checkout produces a
  buildable `apps/web` without a manual extra step.

**CI enforcement:**
- oxlint `react/jsx-no-literals` runs on every PR — zero violations.
- Locale-sync check verifies every non-source locale's keys match
  `messages/en.json` exactly (no missing keys, no orphans).
- `pnpm typecheck` fails if any `m.<key>()` call references a message that
  is not declared in the source catalog.

**Exceptions:**
- Strings that are never user-visible (CSS class names, technical attribute
  values, internal log/event names, identifiers) are exempt.
- Component library internals that receive already-translated strings via
  props are exempt — the responsibility is at the call site.

**Why:** Compile-time message functions make missing or extra translations a
type error, eliminate the runtime overhead of a string-key lookup, and let the
bundler strip messages no shipped code calls. That gives Tanren significantly
smaller per-route bundles and a stronger "all copy is localizable" guarantee
than a runtime translation provider could.
