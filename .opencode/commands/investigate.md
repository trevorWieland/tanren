  # investigate

  ## Purpose

  Diagnose a specific phase failure. For task-scoped failures, record
  root cause and repair guidance so `do-task` can edit the same task.
  For genuinely spec-scoped failures, create follow-up tasks only when
  the source finding is spec-scoped. As a last resort, escalate to a
  blocker for `resolve-blockers`.

  ## Inputs (from your dispatch)

  - The generic check-failure envelope with source phase/check, scope,
    source finding ids, logs, prior attempts, fingerprint, and loop
    index.
  - The bundle index path generated by orchestrator with full,
    untruncated check logs plus a failed-check index for this loop.
  - The diff under suspicion.
  - Prior investigation records for this failure signature (the
    orchestrator uses a root-cause fingerprint to enforce a loop cap,
    default 3).

  ## Responsibilities

  1. Read the failure evidence in full, including every artifact listed
     in the orchestrator bundle index. Distinguish root causes from
     symptoms. Do not modify code; this phase is read-only.
  2. Classify root causes explicitly. Include BDD-specific classes when
     applicable:
     - missing scenario for claimed behavior
     - weak scenario (mutation survivor)
     - behavior-map drift (implementation and mapping out of sync)
     - acceptance criteria ambiguity
     - environment drift
  3. Call `record_investigation_attempt` with the source check,
     fingerprint, source findings, evidence, loop index, and typed root
     causes.
  4. Link root causes to source findings with
     `link_root_cause_to_finding` when source finding ids exist.
  5. For each root cause, choose one action:
     - **Task-scoped failure:** record the root cause and explain the
       same-task repair that `do-task` must perform. If the task's
       description or acceptance criteria are wrong, use `revise_task`
       on that same task.
     - **Spec-scoped failure:** create a follow-up task only with a
       spec-scoped origin such as `SpecInvestigation` or `SpecAudit`.
     - **Cannot resolve autonomously or scope must expand:** call
       `escalate_to_blocker(reason, options)` for `resolve-blockers`.
  6. Add `note` / `question` findings for observations that are not
     immediately actionable but might be useful to the next phase.
  7. Write a narrative for `investigation-report.json`; the durable
     structured attempt history is the event stream.
  8. Call `report_phase_outcome("complete", <one-line summary>)`.

  ## Verification

  If you need to reproduce the failure to ground your diagnosis, use
  the relevant hook: `just check` or
  `just ci` as appropriate. Never modify code.
