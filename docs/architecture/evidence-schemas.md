# Evidence Document Schemas

Every evidence file Tanren produces follows **YAML frontmatter +
markdown body**. Frontmatter is typed, validated by serde, and managed
exclusively via tools; body is free agent prose. This doc is the
authoritative schema spec.

Companion docs:
[orchestration-flow.md](orchestration-flow.md),
[agent-tool-surface.md](agent-tool-surface.md).

---

## 1. File registry

| File | Frontmatter type | Body author | Frontmatter author | Lifecycle |
|---|---|---|---|---|
| `spec.md` | `SpecFrontmatter` | agent (narrative) | tools (`set_spec_*`, `add_spec_*`) | Created in `shape-spec`; append-only additions |
| `plan.md` | `PlanFrontmatter` | generated | orchestrator | Regenerated on every task event |
| `demo.md` | `DemoFrontmatter` | agent (narrative) | tools (`add_demo_step`, `mark_demo_step_skip`, `append_demo_result`) | Steps authored in `shape-spec`; results appended in `run-demo` |
| `audit.md` | `AuditFrontmatter` | agent (narrative) | tools (`record_rubric_score`, `record_non_negotiable_compliance`) | Rewritten per audit cycle |
| `signposts.md` | `SignpostsFrontmatter` | agent (narrative) | tools (`add_signpost`, `update_signpost_status`) | Append + update over spec lifetime |
| `investigation-report.json` | `InvestigationReport` (pure JSON) | generated from tools | tools | Rewritten per investigate cycle |
| `phase-events.jsonl` | — | none | tools (append-only) | Permanent audit trail |

## 2. Schemas

Rust types live in `tanren-domain::methodology::evidence`. All types
derive `Serialize`, `Deserialize`, and where appropriate `JsonSchema`
(from `schemars`). What follows is the canonical shape.

### 2.1 SpecFrontmatter (spec.md)

```yaml
---
kind: spec
spec_id: <uuid-v7>
title: <string>
non_negotiables:
  - <string>
acceptance_criteria:
  - id: <string, kebab-case>
    description: <string>
    measurable: <string>   # How we'll verify
demo_environment:
  connections:
    - name: <string>
      kind: <http|postgres|sqlite|redis|sqs|kafka|fs|custom:…>
      probe: <string>       # How to verify availability
dependencies:
  depends_on_spec_ids: [<uuid-v7>]
  external_issue_refs: [<string>]
base_branch: <string>
touched_symbols:
  - kind: <module|fn|type|trait|macro>
    name: <string>
created_at: <iso-8601>
---
<narrative body: motivation, background, scope conversation>
```

### 2.2 PlanFrontmatter (plan.md)

```yaml
---
kind: plan
spec_id: <uuid-v7>
generated_at: <iso-8601>
---
# Tasks
<rendered task list: id, title, status, guards satisfied, origin>
```

Generated by the orchestrator from `tasks_for_spec(spec_id)`. Read-
only to agents (three-layer enforcement).

### 2.3 DemoFrontmatter (demo.md)

```yaml
---
kind: demo
spec_id: <uuid-v7>
environment:
  probed_at: <iso-8601>
  connections_verified: <bool>
steps:
  - id: <string>
    mode: <RUN|SKIP>
    description: <string>
    expected_observable: <string>
    skip_reason: <string, present iff mode=SKIP>
results:                     # append-only per RUN_DEMO session
  - run_id: <uuid-v7>
    ran_at: <iso-8601>
    step_id: <string>
    status: <pass|fail>
    observed: <string>
---
<narrative walkthrough>
```

### 2.4 AuditFrontmatter (audit.md)

```yaml
---
kind: audit
spec_id: <uuid-v7>
scope: <task|spec>
scope_target_id: <string, present when scope=task>
status: <pass|fail>
fix_now_count: <u32>
rubric:
  - pillar: <string>
    score: <u8, 1..=10>
    rationale: <string>
    supporting_finding_ids: [<uuid-v7>]
non_negotiables_compliance:
  - name: <string>
    status: <pass|fail>
    rationale: <string>
findings: [<uuid-v7>]     # references in Finding projection
generated_at: <iso-8601>
---
<narrative reasoning>
```

### 2.5 SignpostsFrontmatter (signposts.md)

```yaml
---
kind: signposts
spec_id: <uuid-v7>
entries:
  - id: <uuid-v7>
    task_id: <uuid-v7 | null>
    status: <unresolved|resolved|deferred|architectural_constraint>
    problem: <string>
    evidence: <string>
    tried: [<string>]
    solution: <string | null>
    resolution: <string | null>
    files_affected: [<string>]
    created_at: <iso-8601>
    updated_at: <iso-8601>
---
<optional narrative>
```

### 2.6 InvestigationReport (investigation-report.json)

```json
{
  "kind": "investigation",
  "spec_id": "<uuid-v7>",
  "investigation_id": "<uuid-v7>",
  "trigger": {
    "phase": "do-task|task-gate|audit-task|adhere-task|run-demo|spec-gate|audit-spec|adhere-spec",
    "task_id": "<uuid-v7 | null>",
    "error_fingerprint": "<string>",
    "loop_index": <u16>
  },
  "root_causes": [
    {
      "description": "<string>",
      "confidence": "high|medium|low",
      "affected_files": ["<string>"],
      "category": "code_bug|spec_ambiguity|acceptance_criteria_gap|environment_drift|test_gap|external_dependency",
      "suggested_actions": [
        { "kind": "revise_task",  "task_id": "<uuid-v7>", "reason": "<string>" },
        { "kind": "create_task",  "title": "<string>", "description": "<string>", "origin_detail": "<string>" },
        { "kind": "escalate",     "reason": "<string>" }
      ]
    }
  ],
  "narrative": "<string, free prose>",
  "generated_at": "<iso-8601>"
}
```

## 3. Validation lifecycle

### 3.1 On tool call

Every frontmatter-touching tool:
1. Validates the input against its serde schema.
2. Projects the current frontmatter from events.
3. Applies the mutation.
4. Re-renders the frontmatter YAML (deterministic key ordering).
5. Atomically writes (tempfile + rename) the file with updated
   frontmatter and preserved body.
6. Appends the event to `phase-events.jsonl`.

Invalid input → `ToolError::ValidationFailed`; no file changes.

### 3.2 On agent session end

Postflight scans each evidence file:
- Re-parse frontmatter via serde; schema failure →
  `EvidenceSchemaError { file, error, line?, field? }`.
- Diff current body against pre-phase snapshot; non-body edits to
  frontmatter → `UnauthorizedArtifactEdit`; revert.
- Extract narrative body section headers; unknown or missing
  recommended headers → `DocumentationWarning { file, expected }`
  (non-fatal; surfaced in audit rubric under `documentation_complete`).

### 3.3 Error flow

- Validation error during agent session → typed error returned; agent
  may recover or exit.
- Schema error in postflight → phase outcome `Error`; orchestrator
  routes to `investigate`.
- Invalid event on replay → replay fails atomically; no partial
  store mutation.

## 4. Round-trip guarantees

Property-tested in `tanren-domain::methodology::evidence`:

1. `parse(render(frontmatter)) == frontmatter` for every type.
2. `render` produces deterministic output (sorted keys for maps where
   order is not semantic).
3. `parse` tolerates trailing whitespace, BOM, and CRLF.
4. Body is preserved byte-exact through parse + re-render.
5. Unknown frontmatter keys are rejected (strict mode) — prevents
   silent data loss.

## 5. Body conventions

### 5.1 Recognized section headers

Used for `DocumentationWarning` detection (non-fatal heuristics):

- `spec.md`: `## Motivation`, `## Background`, `## Scope`,
  `## Non-Negotiables`, `## Acceptance Criteria`, `## Demo Plan`,
  `## Task Breakdown`
- `demo.md`: `## Environment`, `## Steps`, `## Results`
- `audit.md`: `## Summary`, `## Rubric`, `## Non-Negotiables`,
  `## Findings`, `## Recommendations`
- `signposts.md`: per-entry sub-headers rendered deterministically
  from frontmatter; free narrative allowed at file bottom.

### 5.2 Prose guidance

Agents are encouraged to be terse and specific. Narrative serves
reviewers and downstream tooling; it is not the source of truth.
Structured facts belong in frontmatter.

## 6. File layout on disk

```
{SPEC_ROOT}/
└── {YYYY-MM-DD-HHMM}-{spec_id}-{slug}/
    ├── spec.md
    ├── plan.md                  ← orchestrator-owned
    ├── demo.md
    ├── audit.md
    ├── signposts.md
    ├── investigation-report.json
    ├── phase-events.jsonl        ← committed permanent artifact
    └── visuals/                  ← optional free-form assets
```

`{SPEC_ROOT}` resolves from `tanren.yml` `methodology.variables.
spec_root` with default `tanren/specs`.
