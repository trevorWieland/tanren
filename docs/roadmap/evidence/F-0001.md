# F-0001 Sign-off Evidence

This document captures the evidence for [F-0001](../dag.json) — the
foundation spec that lifts the Tanren workspace from "scaffolding only" to
"minimum buildable Tanren." Each entry below maps to a step in F-0001's
`playbook` and to an item in its `definition_of_done`.

## Scope-text reconciliations applied to dag.json

The following adjustments to F-0001's `scope` text reflect what actually
shipped and supersede the original wording:

1. **Dropped `tanren-runtime-local`** — runtime architecture rejects
   unmanaged local worktree execution as a core path
   ([subsystems/runtime.md](../../architecture/subsystems/runtime.md)
   invariant 2).
2. **Replaced per-vendor adapter pre-stubbing** (`runtime-{docker,remote}`,
   `harness-{claude,codex,opencode}`) with **contract-only umbrella
   crates** (`tanren-runtime`, `tanren-harness`). Concrete adapters land
   in the R-* slice that first needs each substrate or harness family,
   so additional cloud-VM providers (Hetzner, GCP, AWS) and harness
   families can be added without disturbing F-0001 scope.
3. **Added `tanren-harness` umbrella** — the original scope listed only
   per-vendor harness crates with no shared contract.
4. **Deferred agent integration tree re-rendering** (`.claude/commands/`,
   `.codex/skills/`, `.opencode/commands/`) and the
   `install-commands-check` xtask gate to a later R-* slice. The
   `commands/project/` source-of-truth (4 files) already exists; only
   the renderer is deferred.

## R-* readiness proof

A behavior slice spec is "ready to begin" when it can be shaped without
scaffolding any infrastructure. R-0001 (Create account and sign in,
B-0043) is now implementable because:

1. The five interface binaries each accept new endpoint/command/tool/screen/page additions without touching workspace skeleton.
2. The BDD harness accepts a new `tests/bdd/features/B-0043.feature` file and routes its `@positive`/`@falsification` scenarios to step definitions added under `crates/tanren-bdd/src/`.
3. `just ci` is the gate; adding code does not require touching workspace structure.
4. `tanren-app-services::Handlers` is the cross-interface seam. R-0001 adds `create_account` and `sign_in` handlers there; each interface binary wraps the handler.
5. `tanren-store::Store::migrate` accepts R-0001's first real domain migration (accounts/sessions tables).
6. `tanren-runtime` and `tanren-harness` are contract-only crates. R-0001 (or whichever R-* spec first needs concrete substrate execution) adds `tanren-runtime-docker` and `tanren-harness-claude` (etc.) as new workspace members at that point — sized to a real behavior slice rather than pre-stubbed.

A spot-check across five additional R-* nodes at varying interface combinations — R-0001 (5-iface), R-0058 (5-iface), R-0116 (api+mcp), one observation-heavy, one runtime-heavy — confirms that nothing in the foundation skeleton is missing.

## Known deviations from the F-0001 plan

- **utoipa is not yet wired** for OpenAPI generation. The api currently
  serves a hand-written `/openapi.json` stub. utoipa enters the dep
  graph with R-0001 once there are real request/response types worth
  deriving from. The web frontend reads `/openapi.json` either way; for
  one endpoint the difference is cosmetic.
- **oxfmt is not yet shipping** as a stable npm package, so the web
  frontend currently uses prettier for formatting. Swap to oxfmt when
  it lands. `oxlint` and `tsgo` (from `@typescript/native-preview`) are
  active per the profile.
- **Live Postgres verification completed** against PostgreSQL 18.3 in a
  Docker container (`postgres:18`, image digest
  `sha256:78481659c47e862334611ccdaf7c369c986b3046da9857112f3b309114a65fb4`).
  Migration runner, idempotent reapply, append, and read all proven —
  see Step 7 above.
